# pet-loop-orchestrator — 크롬 팻 개발 루프 오케스트레이터

크롬 팻 확장 개발을 3인 전문가 팀으로 조율한다. 핵심은 **"한 루프 = 하나의 검증 가능한 변경"**.
루프가 끝날 때마다 저장소는 항상 `npm run check` 를 통과하는 동작 상태여야 한다.

## 팀 (실행 모드: 에이전트 팀, 3인)
| 에이전트 | 담당 | 검증 |
|---|---|---|
| core-logic-dev | `src/core/` 순수 로직·상태머신 | vitest 단위 테스트 |
| chrome-adapter-dev | `src/chrome/`·background·content·sidepanel | build + E2E |
| harness-qa | 검증 게이트·E2E 하네스·경계면 정합성 | npm run check, test:e2e |

모든 Agent 호출은 `model: "opus"`. 팀 통신은 SendMessage, 작업 공유는 TaskCreate, 산출물은 파일.

## Phase 0: 컨텍스트 확인
1. `.claude/agents/` 3개 정의와 `src/` 구조가 있는지 확인.
2. 요청 유형 판별:
   - 기존 코드 없음/새 기능 → **초기/신규 실행**
   - 기존 기능 수정·부분 재실행 → 해당 담당 에이전트만 재호출
   - 버그 수정 → 재현 케이스 먼저(harness-qa) → 담당자 수정
3. `npm run check` 를 먼저 돌려 현재 그린 상태인지 확인. 레드면 그것부터 고친다.

## Phase 1: 탐색 vs 구현 루프 분리
- **탐색 루프**(코드 수정 없음): "MV3 alarms 최소 주기?" 같은 조사 → 결과를 `docs/notes/` 메모.
- **구현 루프**: 아래 6단계. 탐색이 확정된 뒤에만 시작. 둘을 섞지 않는다.

## Phase 2: 구현 루프 (변경 1건 단위)
```
① 계획   — 이번 루프에서 바꿀 것 1가지를 한 문장으로 선언
② 실패 테스트 — 가능하면 먼저 작성 (core 는 vitest, 통합은 E2E)
③ 최소 구현 — 담당 에이전트가 최소 코드로 통과
④ npm run check — 종합 게이트 실행
⑤ 실패 → 로그 읽고 수정 → ④ (최대 5회)
⑥ 통과 → 커밋(사용자 요청 시) → 다음 루프
```
**루프 규칙:** 한 루프 파일 5개 / 200줄 초과 금지. 넘으면 루프를 쪼갠다.
⑤에서 5회 실패 시 멈추고 원인 분석 + 대안 2가지를 사용자에게 보고. 무한 시도 금지.

## 작업 분배 원칙
- 순수 로직(상태·규칙·계산) → core-logic-dev
- 크롬 API·SW·UI·알림 → chrome-adapter-dev
- 모든 변경의 검증·경계면 확인 → harness-qa (모듈 완성 직후 점진적으로)
- 상태 타입 변경은 core-logic-dev 가 시작하고, chrome-adapter-dev·harness-qa 에 전파한다.

## 데이터 전달
- 조율: TaskCreate/TaskUpdate — 작업 의존(core → adapter → qa) 관리.
- 실시간: SendMessage — 타입 변경 통지, 검증 요청/결과.
- 산출물: 파일(`src/`, `tests/`). 중간 메모는 `docs/notes/`.

## 에러 핸들링
- 에이전트 1회 재시도 후 재실패 → 그 결과 없이 진행하되 보고에 누락 명시.
- 같은 에러 2회 → 접근을 바꾼다(같은 수정 반복 금지).
- 검증 불가능한 변경 → 기능 멈추고 harness-qa 가 하네스부터 보강.

## 완료 정의 (전부 만족해야 완료)
- [ ] `npm run check` 통과
- [ ] E2E 에서 콘솔 에러 0건 (통합 변경 시)
- [ ] 새 로직은 `src/core/` 에 있고 단위 테스트 존재
- [ ] 커밋 메시지에 검증 결과 명시 (커밋은 사용자 요청 시)
- [ ] 하네스로 검증 불가한 변경이면 그 사실 + 수동 확인법 보고

## 테스트 시나리오
**정상 흐름:** "배고픔이 80 넘으면 행복도 2배 감소" 요청 → core-logic-dev 가 실패 테스트+구현 → check 그린 → harness-qa 경계면 확인 → 완료.
**에러 흐름:** E2E 에서 콘솔 에러 1건 발견 → harness-qa 가 재현 케이스로 chrome-adapter-dev 에 수정 요청 → 재검증 → 그린 → 완료.
